# typing-article-app

1. サービス概要

興味関心のある記事を使ってタイピング練習ができるWebアプリです。
プログラミング学習者が、記事の理解とタイピング練習を同時に行えるようにします。

2. このアイデアはどこから生まれたか

2-1. きっかけとなった体験・感情

プログラミング学習中に、タイピング速度の遅さやタイポが原因で
コードを書くスピードや学習効率が落ちていると感じていました。

そのためタイピング練習をしようと既存のサービスを使いましたが、

・意味のない文章
・興味のない内容

ばかりで、練習自体が退屈に感じてしまい継続できないという経験をしました。

また、普段は技術記事を読む習慣があるため、
「どうせならこの時間をタイピング練習にも活かせないか」と感じたことがきっかけです。

2-2. なぜそれが「気になった」のか

自分は、

・学習効率を重視する性格
・無駄な時間を減らしたいという価値観

があり、

「タイピング練習のためだけに時間を取る」ことに違和感を感じました。

また、

・興味のないことは継続しづらい
・意味のある学習でないとモチベーションが続かない

という経験から、
「楽しくない練習は続かない」という課題を強く意識するようになりました。

3. 課題の整理（表面的になっていないか）

3-1. 表に見えている困りごと

・タイピング練習が続かない

3-2. 本当に解決したい課題

なぜ？
→ 興味のない文章で練習しているから

なぜ？
→ タイピング専用の教材は内容に意味・関心がないから

なぜ？
→ 「学習」と「練習」が分断されているから

→ 本当に解決したい課題：
意味のある学習とタイピング練習を同時に行えないこと

4. 想定ユーザーについて
4-1. 想定しているユーザー

年齢設定：
年齢：20〜30代
理由：プログラミング学習やキャリアアップのために技術記事を読む層が多いため

生活・状況設定：
生活・状況：プログラミング学習中で日常的に技術記事を読んでいる
理由：記事を読む時間をそのままタイピング練習に活用できる

環境設定：
環境：PCで学習・記事閲覧を行う
理由：タイピング練習はPC環境で行うのが自然であり、導入ハードルが低い

4-2. 自分とユーザーの距離
・自分自身

理由：
実際に同じ課題を感じているため、ユーザーの行動や感情を具体的に想像できるから

4-3. 実際に使われる可能性について(需要・存在確認)

最初に使う人：プログラミング学習者（自分含む）

利用シーン：記事を読むときに「せっかくだから練習もするか」と思う場面

代替手段：既存のタイピングサイト、通常のブラウザ閲覧

置き換える理由：
興味のある内容なので継続しやすい、習慣化しやすい
同じ時間で「学習＋タイピング」ができるため効率が良い

4-4. ユーザーがサービスを導入して利用するまでのイメージ(導入・継続の流れ)

どんな場面・困りごとがきっかけで、このサービスを知る・思い出すか

・技術記事を読んでいる中でサービスを知る
・SNSでの口コミ

最初の一歩

・URLを貼るだけで使えるため、すぐ試せる

利用を続ける理由

・学習のついでに使える（追加コストがない）
・タイピング結果が可視化され成長を実感できる
・興味のある記事で行うため飽きにくい

5. 既存サービス・競合調査

5-1. 似たサービスの調査
サービス名：e-typing
URL：https://www.e-typing.ne.jp/
どんなことができるか：用意された文章でタイピング練習ができる。スコアやレベルで実力を測ることができる。

サービス名：寿司打
URL：https://sushida.net/
どんなことができるか：制限時間内に表示される単語や文章をタイピングし、スコアを競うゲーム形式のタイピング練習サービス。スピードや正確性を楽しみながら鍛えることができる。


5-2. それでも作りたい理由

既存サービスは、
ゲーム性やスコアでモチベーションを高める工夫はあるが、
文章の内容はあらかじめ用意されたものに限られるため

・実際の学習や知識習得にはつながりにくい
・自分の興味関心と結びつかない

という点に違和感を感じました。


5-3. 差別化

このアプリは、
プログラミング学習中の人が、技術記事を読む際に
タイピング練習も同時にできるアプリです。

実際の学習コンテンツを使うことで、「タイピング練習のための時間」を「学習時間」に変換できます。

6. このサービスで提供したい価値
6-1. ユーザーの変化

・行動：記事を読むだけ → タイピングしながら読む
・気持ち：退屈な練習 → 有意義な学習
・考え方：「練習の時間」ではなく「学習の一部」として捉える

6-2. 価値を一文で

「プログラミング学習者が、記事を読みながらタイピング力を向上できるサービス」


7. このアプリで実現すること

MVPで作る機能
（記事を使ったタイピング体験が成立するかを検証するための最小構成）

・記事テキスト入力（テキストの貼り付け）
・タイピング機能（表示された文章を入力）
・入力文字の正誤判定（1文字単位の簡易判定）
・タイピング結果の表示（速度・ミス数など）


本リリースで作る機能

記事関連
・記事URLからの自動取得（スクレイピング）
・記事本文の自動抽出
・記事の保存・お気に入り管理

タイピング関連
・タイピング履歴の保存
・タイピング結果の分析

その他
・ユーザー登録 / ログイン機能
（タイピング履歴や記事保存などのデータをユーザー単位で管理するため）
・SNS機能
・記事おすすめ機能


8. このアプリの懸念点とその対策
懸念点①
問題：記事の取得が難しい可能性（スクレイピング制約など）
理由：サイトごとに構造が異なるため
対策：
MVPでは「手動入力」も可能にする
対応サイトを限定して実装する

懸念点②
問題：記事の内容がタイピング練習に適さない可能性がある
理由：
コードブロックが多い
記号や英語が多い
レイアウトが崩れる
対策：
本文のみ抽出する
コードブロックは除外する
MVPではシンプルなテキスト記事に限定する

懸念点③
問題：タイピング機能のリアルタイム処理が難しい可能性
理由：入力判定や速度計測など、フロント処理が複雑になるため
対策：
MVPではシンプルなロジックで実装する
ライブラリの利用も検討する
小さく検証してから本実装に進む

懸念点④
問題：リアルタイム入力UIの設計がUXに直結する
理由：1文字遅延・ズレがストレスになるため体験品質に影響する
対策：最初は入力遅延なし・1文字即時反映の最小UIで実装し、体験差を確認する

10. 技術スタック
10-1. 使用予定の技術

フレームワーク：Ruby on Rails 7.2系
言語：Ruby 3.3系
DB：PostgreSQL 16系
デプロイ：Render
ライブラリ：Hotwire / Stimulus

10-2. 技術選定の理由
Ruby on Rails
・RailsはCRUD・認証などを高速に実装できるため
・MVPを素早くリリースするため
・フロントもシンプルに構築できるため

PostgreSQL
・Railsとの相性が良く、標準的な構成であるため
・データ整合性や制約表現が強く、スコア・履歴管理と相性が良いため

Render（必要に応じてFly.ioも検討）
・Railsアプリを最短で公開できるため
・DB（PostgreSQL）も含めて構築が簡単なため
・MVP段階では「開発速度と検証」を優先するため
・まずはRenderでリリースし、必要に応じてFly.ioへ移行検討する

ライブラリ：Hotwire / Stimulus
・Rails標準の思想に近く、開発速度を重視したMVP開発に適しているため
・タイピング処理のような軽量なインタラクション（入力監視・スコア更新）に対して十分な表現力を持つため
・フロントエンドを過度に複雑化せず、Rails中心の設計を維持できるため
・MVP段階ではReactなどの導入よりも、検証速度を優先するため
（→初期段階ではReactのような複雑なフロント構成は避け、Rails中心で検証速度を優先する）

10-3. キャッチアップ不足の懸念
JS部分（タイピング処理）の実装
・1文字単位の入力監視と正誤判定
・タイピング速度（WPM）のリアルタイム計算
・DOM更新のタイミング設計（過剰レンダリング防止）
